Code-Anchored Context

Two practices, one principle

Keep truth anchored in the repository, close to the code. Two independent practices solve two different problems: Durable Context owns the working bench and decision log while you plan and build; Code-Anchored Docs keeps shipped-behavior documentation accurate from release tags. Adopt either or both.

Two Problems, Not One

Coding agents make both problems visible faster. They are related but distinct — and each has its own practice.

Reasoning escapes while you build

Plans live in sessions, not in the repo. Decisions evaporate when the chat closes. Work cannot be handed off. The next agent starts from scratch.

→ Durable Context

Docs drift after you ship

Product documentation lives apart from the code. Nobody wants to write it, and it stops matching what the system actually does the moment people stop maintaining it.

→ Code-Anchored Docs

The Two Practices

No hard dependency between them. Install either package, read either series, use either workflow on its own.

Durable Context

Own the working bench (context/) and the decision log (decisions/). Plan durably, promote accepted decisions, archive the bench. What you plan becomes the agent's working context when implementation starts.

npx @code-anchored-context/planning init

Code-Anchored Docs

Keep shipped-behavior docs accurate from release tags. Refresh from the diff between tags once per release. The code diff is what you inspect; judgment decides what changed in released behavior.

npx @code-anchored-context/reference init

Keep truth as close to code as possible, and keep it structured enough that both humans and agents can find it.
